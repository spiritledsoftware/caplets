# Cloud runtime provider feasibility — documentary pre-gate

- **Date:** 2026-07-11
- **Scope:** The planned portable full runtime: AWS Application Load Balancer (ALB) → one ECS/Fargate Node task in private networking with external durable stores; Cloudflare Worker → deterministic Durable Object (DO) → one Container, with a deployment-scoped repository bridge to Worker-owned D1/R2/secrets; and Alchemy **0.93.12** custom AWS resources.
- **Method:** Primary sources only. AWS and Cloudflare findings are from their official documentation. Alchemy findings are from the installed `alchemy` **0.93.12** package, cross-checked against the project’s first-party `v0.93.12` source tag.
- **Meaning of status:** **Documented** means the linked primary source states the capability/limit. **Inference** is a design conclusion from documented behavior. **Contradicted** means the literal planned claim is not provided by the cited API/source. **Live proof required** means documentation cannot establish the end-to-end behavior, permissions, timing, or recovery property.

> **U10 is not passed by this note.** The plan defines U10 as a _credentialed, real-provider_ architecture gate and explicitly requires both real-account gates to pass before U1/U2. Documentary evidence can narrow the experiment and identify blockers; it cannot demonstrate provider-account entitlements, exact routing, protocol behavior, lifecycle timing, IAM, or cleanup in a real deployment.
>
> **No credentials or provider commands were used for this research.** Every provisional conclusion below remains subject to the residual live tests at the end.

## Executive finding

There is **no documentary impossibility** for the AWS compute/ingress shape or the Cloudflare Worker/DO/Container shape. The provider docs directly support the relevant ingress, WebSocket, ephemeral-disk, external-storage, and scoped Worker-binding mechanisms. Two load-bearing qualifications prevent a documentary “go”:

1. **Alchemy 0.93.12 does not expose a separate custom-resource `read` handler.** Its custom `Resource` handler is dispatched only with `create`, `update`, or `delete` context; Alchemy’s internal `read` phase returns the persisted state without calling the handler. Thus a literal CloudFormation-style CRUD/read hook is **Contradicted**. A custom AWS resource can still call AWS SDK `Describe*` operations itself during create/update/delete/retry, but must define that reconciliation contract.
2. **Alchemy persists `creating` before executing a handler and only writes final output after success.** A process failure after an AWS create call can leave persisted state without the cloud identifier. Convergence and orphan cleanup therefore require resource-owned discovery using deterministic names/tags/client tokens, idempotent AWS requests, and a carefully specified retained-data policy. Alchemy’s built-in orphan pass only destroys resources that remain in its state store; it is not a cloud-account inventory/reconciler.

The Cloudflare repository bridge has a documented least-privilege substrate: Container outbound handlers run in the Workers runtime, have bindings, can scope by `ctx.containerId`, and can keep secrets outside the Container. The exact bridge authentication, fencing, transaction/error mapping, and streamed response behavior remain **Live proof required**.

## AWS: ALB → one private ECS/Fargate Node task

### Load-bearing claims

| ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AWS-1 | ALB listeners support HTTP/HTTPS; ALB provides native WebSocket support. An HTTP/1.1 upgrade becomes a persistent WebSocket connection between client and target through the ALB. Source: [AWS ALB listener documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-listeners.html#listener-configuration).                                                                                                                                                                                                                                                  | **Documented**          | An ALB can front the Node HTTP/WebSocket surface without an NLB solely for WebSockets. | AWS-2 | ALB closes a connection with no data sent/received for the configured idle timeout. Default is 60 seconds; valid configured range is 1–4,000 seconds. AWS recommends sending at least one byte before idle expiry for lengthy operations and setting the application timeout higher than ALB’s. HTTP/2 PING does not reset this timer. Client HTTP keepalive is separately bounded (default one hour; range 60–604,800 seconds). Source: [ALB connection attributes](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-load-balancer-attributes.html#connection-idle-timeout). | **Documented**          | Long-lived HTTP streams and quiet WebSockets must emit application-level traffic/heartbeats and must choose ALB idle/keepalive values deliberately. “Long-lived” is bounded by these attributes and reconnect behavior, not unlimited persistence.                                                                                                                      |
| AWS-3 | Each Fargate task gets a managed ENI with a private primary IP. A public IP is optional only in a public subnet. Fargate services support ALB/NLB and require `ip`, not `instance`, target groups. Source: [Fargate task networking](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-task-networking.html).                                                                                                                                                                                                                                                                       | **Documented**          | Public ALB subnets plus private task subnets with an ALB `ip` target group match the documented Fargate networking model. No public IP is required on the runtime task.                                                                                                                                                                                                 |
| AWS-4 | ECS rolling deployments replace running tasks. `minimumHealthyPercent` and `maximumPercent` determine how many tasks may be stopped/started; when maximum permits, ECS launches unhealthy replacements before stopping unhealthy tasks. Source: [ECS rolling deployment](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/deployment-type-ecs.html).                                                                                                                                                                                                                                       | **Documented**          | Physical task overlap is possible even when desired count is one (for example, suitable maximum percent). The planned single _logical_ instance therefore needs its own durable lease/fencing before either task mutates state.                                                                                                                                         |
| AWS-5 | Linux Fargate task storage is explicitly **ephemeral**: platform 1.4.0+ provides 20 GiB by default (configurable to 200 GiB) for task storage/bind mounts. Source: [Fargate ephemeral storage](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-task-storage.html).                                                                                                                                                                                                                                                                                                                | **Documented**          | `/data`-style local runtime state cannot be the restart/redeploy durability boundary. It must be materialized from and committed to SQL/object/key services; ephemeral disk is only a cache/workspace.                                                                                                                                                                  |
| AWS-6 | A private-subnet Fargate task needs a route to pull images: either NAT or an ECR interface VPC endpoint. On Linux 1.4.0+, image pulls, logs, secret retrieval, and application traffic use the task ENI; AWS lists ECR credentials/image pulls/logs/Secrets Manager or SSM in the task networking table. Source: [Fargate task networking and traffic table](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-task-networking.html).                                                                                                                                               | **Documented**          | The default private network must provide egress for image pull and all runtime dependencies. Use private endpoints where available (at least ECR is explicitly documented here); provide NAT for any outbound service not reachable by an endpoint. The exact endpoint set and policies are region/account/application dependent and require a live IAM/flow-log proof. |
| AWS-7 | The sources establish HTTP and WebSocket capability but do not prove this repository’s Node server will preserve chunk timing, authentication, upgrade headers, dashboard assets, attach/control, and reconnect semantics through a configured ALB/Fargate target group.                                                                                                                                                                                                                                                                                                                              | **Live proof required** | U10 must run sustained authenticated chunk streaming and WebSocket traffic through the actual public origin, including an idle period/heartbeat and a task replacement.                                                                                                                                                                                                 |

### AWS topology and durable-state conclusion

**Inference:** The requested ALB/public-subnet → private Fargate/private-store topology is consistent with AWS’s documented networking model. Its correctness is not implied by the topology diagram: it requires a live account proof of route tables, ALB-to-task security-group edge, task-to-store/key edges, ECR/log/secrets connectivity, denied direct task/SQL ingress, and narrow task role permissions. The durable-state design is necessary rather than optional because Fargate disk is documented as ephemeral.

**Provisional AWS provider verdict: _GO, conditional on U10 live proof_.** There is no documented protocol or topology blocker, but no credential-free source can prove account capacity, availability-zone placement, IAM, endpoint support, actual stream behavior, or lease safety during overlap.

## Alchemy 0.93.12 custom AWS resources

### Version and available surface

The installed lockfile pins `alchemy` to **0.93.12**. Its package metadata identifies that exact version and repository; the tagged first-party source is [Alchemy `v0.93.12`](https://github.com/alchemy-run/alchemy/tree/v0.93.12). The AWS EC2 entrypoint exports VPC, subnet, route table, route, NAT gateway, internet gateway, and security-group primitives ([tagged source](https://github.com/alchemy-run/alchemy/blob/v0.93.12/alchemy/src/aws/ec2/index.ts)). The public AWS entrypoint exports that EC2 surface and other AWS primitives, but not first-party ECS, ALB/ELBv2, ECR, RDS, ACM, or Route 53 constructs ([tagged source](https://github.com/alchemy-run/alchemy/blob/v0.93.12/alchemy/src/aws/index.ts)). Custom AWS resources are therefore required for those service APIs.

### Exact first-party Alchemy API signatures

```ts
export function Resource<const Type extends string, F extends ResourceLifecycleHandler>(
  type: Type,
  fn: F,
): Handler<F>;

export type ResourceLifecycleHandler = (
  this: Context<any, any>,
  id: string,
  props: any,
) => Promise<ResourceAttributes>;

export interface CreateContext<Out extends ResourceAttributes> extends BaseContext<Out> {
  phase: "create";
  output?: undefined;
  props?: undefined;
}

export interface UpdateContext<
  Out extends ResourceAttributes,
  Props extends ResourceProps = ResourceProps,
> extends BaseContext<Out> {
  phase: "update";
  output: Out;
  props: Props;
}

export interface DeleteContext<
  Out extends ResourceAttributes,
  Props extends ResourceProps = ResourceProps,
> extends BaseContext<Out> {
  phase: "delete";
  output: Out;
  props: Props;
}
```

These are copied from Alchemy 0.93.12 [`resource.ts`](https://github.com/alchemy-run/alchemy/blob/v0.93.12/alchemy/src/resource.ts) and [`context.ts`](https://github.com/alchemy-run/alchemy/blob/v0.93.12/alchemy/src/context.ts). There is intentionally no `ReadContext` in that union.

End users cannot send non-HTTP TCP/UDP directly to a Container. Use a deployment-scoped name and set container maximum instances to one; deterministic naming alone does not implement mutation fencing.                                                                                                              |
| CF-3 | Cloudflare’s Container example states that WebSocket requests are automatically forwarded when the Worker returns `getContainer(...).fetch(request)`. | CF-4 | Workers support HTTP response streaming: HTTP-triggered Worker duration has no hard limit while the client remains connected, and Cloudflare specifically recommends streaming rather than buffering large bodies. Source: [Workers limits: duration and memory](https://developers.cloudflare.com/workers/platform/limits/).                                                                                                                                                                                                                                                                                                                         | **Documented**          | The Worker layer supports streaming. The Container routing docs say `containerInstance.fetch(request)` passes requests and returns a `Response`, but do not explicitly promise chunk-flush/cadence preservation for this exact Node → Container → DO → Worker chain. End-to-end streamed Node behavior is therefore separately live-gated. |
| CF-5 | A Container runs in its own VM. When it sleeps/restarts, its disk is fresh from the image; all disk is ephemeral. Cloudflare may route a restarted instance to a different location. Shutdown delivers `SIGTERM`, then `SIGKILL` after 15 minutes. Source: [Container lifecycle architecture](https://developers.cloudflare.com/containers/platform-details/architecture/#persistent-disk).                                                                                                                                                                                                                                                           | **Documented**          | All required state must restore through the bridge; local materialization is disposable. U10 must prove an intentional restart does not lose required configuration/workspace data.                                                                                                                                                        |
| CF-6 | Internet egress is allowed by default. `enableInternet = false` makes egress deny-by-default except allowed hosts/outbound handlers; only ports 80, 443, and DNS are available in that setting. Outbound handlers can run trusted Worker code, and `outboundByHost` maps a virtual hostname to it. Source: [Container outbound traffic](https://developers.cloudflare.com/containers/platform-details/outbound-traffic/).                                                                                                                                                                                                                             | **Documented**          | Set `enableInternet = false` and allow only the repository bridge plus explicitly required upstreams. This is the documented network control for a constrained Container.                                                                                                                                                                  |

Source: [Container connections to Workers/bindings](https://developers.cloudflare.com/containers/platform-details/workers-connections/) and [outbound-handler secret injection](https://developers.cloudflare.com/containers/platform-details/outbound-traffic/#securely-inject-credentials). | **Documented**          | This is a documented substrate for a Worker-hosted, least-privilege D1/R2/key repository bridge. Prefer it to passing account tokens or D1/R2 credentials into the Node container.                                                                                                                                                         |
| CF-8 | Cloudflare documents transport to a virtual hostname and `ctx.containerId`, but not an application-level protocol providing the plan’s rotating bridge secret, deployment/fencing-generation authorization, operation allowlist, idempotency, transaction-to-error mapping, or timeout semantics.                                                                                                                                                                                                                                                                                                                                                     | **Live proof required** | The bridge must be implemented as explicit Worker code, not inferred from the binding. It must bind every request to the deterministic Container/DO and current deployment/fencing generation, reject cross-deployment/replayed operations, and be attacked/tested from the Container.                                                     |

### Cloudflare bridge conclusion

**Inference:** A secure bridge is feasible with no account-wide Cloudflare credential inside the Container: use a private virtual hostname/outbound handler, deny general internet by default, derive the Container identity from `ctx.containerId`, and let Worker code access bindings. The security design must not mistake the virtual hostname for authentication. The documented substrate protects Worker-held secrets from being passed into the sandbox, but the planned authorization/fencing protocol remains application code and must be live-tested.

**Provisional Cloudflare provider verdict: _GO, conditional on U10 live proof_.** No Cloudflare documentation contradicts the routing or binding boundary. The architecture is blocked from implementation progression until a real paid account proves the exact bridge, forwarding/stream timing, one-instance/restart behavior, and entitlement limits.

### Documentary blocker/ambiguity register

1. **Alchemy custom-resource `read` lifecycle — documentary blocker for the literal claim.** The exact source has no `ReadContext`, and read phase bypasses the handler. Resolve by changing the custom-resource contract to create/update/delete plus explicit AWS discovery/reconciliation—not by assuming an unavailable lifecycle hook.
2. **Cloudflare Container-to-Worker bridge authentication/fencing — documentary ambiguity.** The documented outbound handler is sufficient to keep bindings/secrets outside the Container, but Cloudflare does not provide the plan’s repository authorization protocol. Resolve only with implemented protocol tests in a real account.
3. **Cloudflare streamed Node response propagation — documentary ambiguity.** Worker streaming and Container WebSocket forwarding are documented; the exact stream cadence/backpressure behavior across `Container.fetch()` is not stated. Resolve with an external, authenticated timed-chunk test.
4. **Cloudflare key management — documentary ambiguity.** Worker secrets supply encrypted text bindings with 5 KB variable limits, but the reviewed primary sources do not document a general Cloudflare KMS/envelope-key/version-rewrap service. The implementation must either use a documented supported secret/key design within these limits or stop and obtain a separately verified provider primitive; U10 must validate the chosen design without exposing plaintext/key material.
5. **AWS private egress/IAM — live-account dependent.** AWS documents task ENI traffic and ECR private endpoints, but the complete dependency endpoint matrix, endpoint policies, NAT need, region support, and least-privilege IAM are architecture- and account-specific.
